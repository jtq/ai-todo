# Spec 002: Task Comments

Created: 2026-06-12 11:32 Europe/London

## 1. Purpose

Add task-owned comments for detailed progress updates and additional context.

Comments are backend API resources. They are not a frontend feature, notification feature, or collaboration system.

## 2. Domain Model Changes

### Task

Add an optional `comments` field to `Task`.

```ts
interface Task {
  comments: EntityId[];
}
```

Rules:

- `comments` is an ordered array of comment IDs owned by the task.
- If no comments exist, API responses should return an empty array.
- Deleting a task must delete its comments.

### Comment

```ts
interface Comment {
  id: EntityId;
  taskId: EntityId;
  body: string;
  position: number;
  createdAt: IsoDateTime;
  updatedAt: IsoDateTime;
}
```

Rules:

- `id` is generated automatically from the global entity ID sequence.
- `taskId` is mandatory and references the task that owns the comment.
- `body` is required markdown text.
- `position` controls the comment's order within its owning task.
- Tasks and comments have a one-to-many relationship.
- Each comment belongs to exactly one task.
- Comments must not be shared by multiple tasks.
- Deleting a task must delete its comments.

## 3. Database Migration

Add a new numbered migration:

```text
003_add_task_comments.ts
```

The migration must:

- Allow `comment` as an entity type in the `entities` table.
- Create a `comments` table.
- Add an index for ordered task comments.

Suggested table:

```sql
create table comments (
  id text primary key references entities(id) on delete cascade,
  task_id text not null references tasks(id) on delete cascade,
  body text not null,
  position integer not null,
  created_at text not null,
  updated_at text not null,
  unique (task_id, position)
);
```

Suggested index:

```sql
create index idx_comments_task_order on comments(task_id, position);
```

## 4. Public API

### Comments

```http
POST /api/v1/tasks/:id/comments
GET /api/v1/comments/:id
PATCH /api/v1/comments/:id
DELETE /api/v1/comments/:id
```

Create comment request:

```ts
interface CreateCommentRequest {
  body: string;
}
```

Update comment request:

```ts
interface UpdateCommentRequest {
  body?: string;
}
```

### Task Comment Ordering

```http
PATCH /api/v1/tasks/:id/comments/order
```

Comment order request:

```ts
interface ReorderTaskCommentsRequest {
  commentIds: EntityId[];
}
```

Rules:

- Creating a comment appends it to the end of its owning task's comment list.
- Removing a comment preserves the order of remaining comments.
- Reordering must include exactly the same comment IDs already owned by the task.
- Reordering must reject comment IDs owned by other tasks.

## 5. Testing Requirements

Tests must cover:

- Comment creation.
- Comment retrieval.
- Comment updates.
- Comment deletion.
- Ordered task comments on task responses.
- Comment reorder validation.
- Enforcement that each comment belongs to exactly one task.
- Rejection of attempts to reorder or reference comments owned by another task.
- Missing task errors when creating or reordering comments.
- Missing comment errors when retrieving, updating, or deleting comments.
- Validation errors for empty comment bodies and duplicate reorder IDs.
- OpenAPI coverage for comment endpoints and schemas.
