# Spec 001: AI-Operable Todo List And Task Tracker Backend

Created: 2026-06-03 19:46 Europe/London

## 1. Purpose

Build a TypeScript backend service for a todo list and task tracker that can be discovered, queried, and operated on by external clients, including AI systems.

The project consists entirely of backend APIs, domain logic, and storage/persistence systems. It must not include a frontend, web UI, CLI, MCP server, desktop UI, or other client-facing interface beyond backend APIs. Separate projects will consume this backend from web apps, CLIs, MCP servers, automations, external AI systems, or other integrations.

## 2. Technology Requirements

Use:

- TypeScript
- Node.js 24.x LTS, targeting the latest available LTS patch at implementation time
- npm, using the default npm version bundled with the targeted Node.js 24.x LTS release
- ESM modules
- Strict TypeScript compiler settings
- A lightweight embedded relational database for persistence
- JSON over HTTP for the public API

Recommended implementation stack:

- Runtime: Node.js 24.x LTS
- Package manager: npm bundled with the targeted Node.js 24.x LTS release
- HTTP framework: Fastify
- Validation: Zod
- API documentation: OpenAPI 3.1
- Persistence: SQLite, using a TypeScript-friendly driver such as `better-sqlite3`
- Tests: Vitest
- Formatting/linting: Prettier and ESLint

The persistence implementation must be isolated behind repository interfaces so the storage engine can be replaced later without changing route handlers or business services.

## 4. Global Entity IDs

All persisted entities in the system must have a globally unique, incrementing, auto-generated alphanumeric ID string.

Requirements:

- One global ID sequence is shared by every entity type.
- IDs are generated by the backend only.
- Clients must not provide IDs for new entities.
- IDs are never reused, including after deletion.
- IDs are incrementing according to the underlying numeric sequence.
- IDs are encoded as Base62 using characters `0-9A-Za-z`.
- IDs are case-sensitive.
- IDs should be kept as short as possible for typing convenience.
- ID generation must be atomic.

Example sequence:

```text
1 -> 1
9 -> 9
10 -> A
35 -> Z
36 -> a
61 -> z
62 -> 10
3843 -> zz
3844 -> 100
```

Suggested storage record:

```text
id_sequence(name text primary key, value integer not null)
```

## 5. Domain Model

### 5.1 Shared Types

```ts
type EntityId = string;

type IsoDateTime = string;
type IsoDate = string;

type Deadline =
  | {
      kind: "date";
      date: IsoDate;
    }
  | {
      kind: "datetime";
      datetime: IsoDateTime;
    };
```

All server-side datetimes must be stored and returned as UTC ISO 8601 strings. UTC datetime strings should use the `Z` suffix, for example `2026-06-03T18:46:00.000Z`.

The backend must not store local timezone offsets for datetime fields. Conversion to and from local timezones for display, editing, or user preference handling is the exclusive responsibility of client applications.

Date-only deadlines represent calendar dates without a time component. Date-only values are stored as `YYYY-MM-DD` and are not converted by the backend. Datetime deadlines represent exact UTC instants.

### 5.2 Task

```ts
type TaskStatus = "draft" | "todo" | "in_progress" | "on_hold" | "completed" | "wont_do";

type ProgressTracker = "computed_from_subtasks" | "manual";

interface Task {
  id: EntityId;
  title: string;
  description?: string;
  status: TaskStatus;
  createdAt: IsoDateTime;
  completedAt?: IsoDateTime;
  deadline?: Deadline;
  attachments: EntityId[];
  progressTracker: ProgressTracker;
  progress: number;
  parentTaskId?: EntityId;
  childTaskIds: EntityId[];
  blockedByTaskIds: EntityId[];
  createdBy?: string;
  updatedAt: IsoDateTime;
}
```

Task field rules:

- `id` is generated automatically.
- `title` is required plain text.
- `description` is optional markdown.
- `status` must be one of `draft`, `todo`, `in_progress`, `on_hold`, `completed`, or `wont_do`.
- `on_hold` represents the display status "On hold".
- `wont_do` represents the display status "Won't do".
- `createdAt` defaults to the current server time in UTC.
- `completedAt` is optional.
- `deadline` is optional and may be date-only or datetime.
- `attachments` is an ordered array of attachment IDs owned by this task.
- `progressTracker` is either `computed_from_subtasks` or `manual`.
- `progress` is a floating point value in the inclusive range `0` to `1`.
- `progress` defaults to `0`.
- `parentTaskId` optionally references the single parent task of this task.
- `childTaskIds` references zero or more child tasks.
- `blockedByTaskIds` references zero or more tasks currently blocking this task.

Relationship rules:

- Parent and child references are reciprocal.
- Adding a child task to a parent must also add the parent to the child.
- Adding a parent task to a child must also add the child to the parent.
- A task may have at most one parent task.
- Setting a new parent for a task replaces any existing parent relationship for that task.
- Removing either side of a parent/child relationship must update both tasks.
- Parent/child relationships must not create cycles.
- A task cannot be its own parent or child.
- A task cannot block itself.
- A task cannot be blocked by a non-existent task.
- Deleting a task must remove references to it from other tasks.

Status rules:

- Default status is `todo`, unless explicitly created as `draft`.
- Tasks may transition from any valid status to any other valid status.
- The backend must not enforce a workflow order between statuses.
- When status becomes `completed`, `completedAt` should default to the current server time in UTC if not supplied.
- When status changes away from `completed`, `completedAt` should be cleared unless an explicit administrative update keeps it.

### 5.3 Attachment

Attachments represent files or URLs to resources related to a task. All attachment targets are represented as URLs. Local files must use the `file://` protocol.

```ts
interface Attachment {
  id: EntityId;
  taskId: EntityId;
  description: string;
  url: string;
  type?: string;
  position: number;
  createdAt: IsoDateTime;
  updatedAt: IsoDateTime;
}
```

Attachment field rules:

- `id` is generated automatically.
- `taskId` is mandatory and references the task that owns the attachment.
- `description` is markdown text.
- `url` is mandatory.
- Local file attachments must be represented with a `file://` URL.
- Remote resource attachments must use the resource URL directly.
- `type` is optional and should contain the MIME type of the URL resource when known.
- `position` controls the attachment's order within its owning task.
- Tasks and attachments have a one-to-many relationship.
- Each attachment belongs to exactly one task.
- Attachments must not be shared by multiple tasks.
- Deleting a task must delete its attachments.

## 6. Progress Behavior

### 6.1 Manual Progress

When `progressTracker` is `manual`:

- Clients may set `progress`.
- `progress` must be between `0` and `1`.
- Completing a task may set progress to `1`, unless the client explicitly supplies another valid value.

### 6.2 Computed Progress

When `progressTracker` is `computed_from_subtasks`:

- Clients must not directly set `progress`.
- `progress` is computed from direct child tasks.
- If the task has no child tasks, progress is `1` when the task is completed and `0` otherwise.
- If the task has child tasks, progress is the average progress of direct child tasks.
- Completed child tasks count as progress `1`.
- Progress changes to a child task must update computed ancestors recursively.

Initial formula:

```text
task.progress = average(child.progress for each direct child)
```

If a more sophisticated weighted progress model is needed, it should be introduced by a later spec.

## 7. Persistence Design

The database must use a relational schema. SQLite is the recommended store because it is lightweight, embedded, durable, and does not require external infrastructure.

Relational storage requirements:

- Use foreign keys where relationships are represented directly.
- Enable SQLite foreign key enforcement on every connection with `PRAGMA foreign_keys = ON`.
- Use transactions for multi-record updates.
- Store every datetime value as a UTC ISO 8601 string with a `Z` suffix.
- Store date-only deadlines separately from datetime deadlines.
- Store task relationship edges in join tables rather than duplicating them as independent records.
- Store task attachment ownership and order on attachment records.

Suggested tables:

```sql
create table id_sequence (
  name text primary key,
  value integer not null
);

create table entities (
  id text primary key,
  entity_type text not null check (entity_type in ('task', 'attachment')),
  sequence_value integer not null unique,
  created_at text not null
);

create table tasks (
  id text primary key references entities(id) on delete cascade,
  title text not null,
  description text,
  status text not null check (status in ('draft', 'todo', 'in_progress', 'on_hold', 'completed', 'wont_do')),
  created_at text not null,
  completed_at text,
  deadline_kind text check (deadline_kind in ('date', 'datetime')),
  deadline_date text,
  deadline_datetime text,
  progress_tracker text not null check (progress_tracker in ('computed_from_subtasks', 'manual')),
  progress real not null default 0 check (progress >= 0 and progress <= 1),
  created_by text,
  updated_at text not null,
  check (
    (deadline_kind is null and deadline_date is null and deadline_datetime is null)
    or (deadline_kind = 'date' and deadline_date is not null and deadline_datetime is null)
    or (deadline_kind = 'datetime' and deadline_datetime is not null and deadline_date is null)
  )
);

create table attachments (
  id text primary key references entities(id) on delete cascade,
  task_id text not null references tasks(id) on delete cascade,
  description text not null,
  url text not null,
  type text,
  position integer not null,
  created_at text not null,
  updated_at text not null,
  unique (task_id, position)
);

create table task_relationships (
  parent_task_id text not null references tasks(id) on delete cascade,
  child_task_id text not null references tasks(id) on delete cascade,
  primary key (parent_task_id, child_task_id),
  check (parent_task_id <> child_task_id)
);

create table task_blocks (
  blocked_task_id text not null references tasks(id) on delete cascade,
  blocking_task_id text not null references tasks(id) on delete cascade,
  primary key (blocked_task_id, blocking_task_id),
  check (blocked_task_id <> blocking_task_id)
);
```

Suggested indexes:

```sql
create index idx_tasks_status on tasks(status, id);
create index idx_tasks_created_at on tasks(created_at, id);
create index idx_tasks_deadline_date on tasks(deadline_date, id);
create index idx_tasks_deadline_datetime on tasks(deadline_datetime, id);
create index idx_attachments_task_order on attachments(task_id, position);
create index idx_task_relationships_child on task_relationships(child_task_id, parent_task_id);
create index idx_task_blocks_blocking on task_blocks(blocking_task_id, blocked_task_id);
```

Repository responsibilities:

- Generate IDs atomically.
- Create, update, and delete table records.
- Rely on relational constraints for referential integrity where possible.
- Perform relationship updates atomically in transactions.
- Provide cursor-based list operations.
- Hide database-specific details from services and route handlers.

## 8. Public API

Expose a JSON HTTP API under:

```text
/api/v1
```

### 8.1 Health And Metadata

```http
GET /health
GET /openapi.json
```

`GET /health` returns service and persistence health.

`GET /openapi.json` returns the OpenAPI document for the backend API.

### 8.2 Tasks

```http
POST /api/v1/tasks
GET /api/v1/tasks
GET /api/v1/tasks/:id
PATCH /api/v1/tasks/:id
DELETE /api/v1/tasks/:id
```

Create task request:

```ts
interface CreateTaskRequest {
  title: string;
  description?: string;
  status?: TaskStatus;
  deadline?: Deadline;
  progressTracker?: ProgressTracker;
  progress?: number;
  parentTaskId?: EntityId;
  childTaskIds?: EntityId[];
  blockedByTaskIds?: EntityId[];
}
```

Update task request:

```ts
interface UpdateTaskRequest {
  title?: string;
  description?: string | null;
  status?: TaskStatus;
  completedAt?: IsoDateTime | null;
  deadline?: Deadline | null;
  progressTracker?: ProgressTracker;
  progress?: number;
  blockedByTaskIds?: EntityId[];
}
```

Task list query parameters:

```text
status
parentTaskId
childTaskId
blockedByTaskId
hasDeadline
deadlineBefore
deadlineAfter
createdBefore
createdAfter
search
limit
cursor
sort
```

Supported task list sort values:

```text
created_at_asc
created_at_desc
deadline_asc
deadline_desc
title_asc
status_asc
```

### 8.3 Task Relationships

```http
PUT /api/v1/tasks/:id/parent/:parentTaskId
DELETE /api/v1/tasks/:id/parent/:parentTaskId

PUT /api/v1/tasks/:id/children/:childTaskId
DELETE /api/v1/tasks/:id/children/:childTaskId

PUT /api/v1/tasks/:id/blocked-by/:blockingTaskId
DELETE /api/v1/tasks/:id/blocked-by/:blockingTaskId
```

Relationship endpoints must:

- Validate that all referenced tasks exist.
- Reject self-relationships.
- Reject parent/child cycles with `409 Conflict`.
- Update reciprocal parent/child references.
- Setting a parent replaces any previous parent for that task.
- Recompute affected progress values when child relationships change.

### 8.4 Attachments

Attachments are task-owned. Each attachment belongs to exactly one task and cannot be shared across tasks.

```http
POST /api/v1/tasks/:id/attachments
GET /api/v1/attachments/:id
PATCH /api/v1/attachments/:id
DELETE /api/v1/attachments/:id
```

Create attachment request:

```ts
interface CreateAttachmentRequest {
  description: string;
  url: string;
  type?: string;
}
```

`url` is mandatory when creating an attachment. The owning task is identified by the `:id` path parameter on `POST /api/v1/tasks/:id/attachments`.

Update attachment request:

```ts
interface UpdateAttachmentRequest {
  description?: string;
  url?: string;
  type?: string | null;
}
```

### 8.5 Task Attachment Ordering

```http
PATCH /api/v1/tasks/:id/attachments/order
```

Attachment order request:

```ts
interface ReorderTaskAttachmentsRequest {
  attachmentIds: EntityId[];
}
```

Rules:

- Creating an attachment appends it to the end of its owning task's attachment list unless an explicit valid position is supplied by a future API revision.
- Removing an attachment preserves the order of remaining attachments.
- Reordering must include exactly the same attachment IDs already owned by the task.
- Reordering must reject attachment IDs owned by other tasks.

## 9. External AI Operability

The backend must expose clear, complete, machine-readable APIs that external systems can use. External AI systems may discover, query, and operate on tasks and attachments through the public API.

Requirements:

- The OpenAPI document must describe every public endpoint, request body, response body, query parameter, validation rule, and error shape.
- Resource schemas must use stable field names and explicit enum values.
- API responses must include enough relationship data for external clients to understand parent/child links, blocking relationships, and task-owned attachments.
- Mutating APIs must be deterministic and validation-heavy so external automation clients can safely operate the task store.

## 10. Error Handling

All API errors should use this shape:

```ts
interface ApiError {
  error: {
    code: string;
    message: string;
    details?: unknown;
  };
}
```

Common status codes:

- `200 OK`
- `201 Created`
- `204 No Content`
- `400 Bad Request`
- `404 Not Found`
- `409 Conflict`
- `422 Unprocessable Entity`
- `500 Internal Server Error`

Validation failures should return `422`.

Relationship conflicts, such as cycles or self-blocking, should return `409`.

Missing entities should return `404`.

Datetime validation requirements:

- Incoming datetime values must either already be UTC ISO 8601 strings ending in `Z`, or be rejected with `422`.
- The backend must not infer a timezone for naive datetime strings.
- The backend must not accept local datetime strings such as `2026-06-03T10:30:00` for datetime fields.
- The backend must return all datetime fields in UTC.
- Date-only fields must remain date-only strings and must not be converted to UTC datetimes.

## 14. Acceptance Criteria

The implementation is acceptable when:

- The backend starts on Node.js 24.x LTS.
- The service exposes `/health`.
- The service exposes `/openapi.json`.
- Tasks can be created, listed, retrieved, updated, and deleted.
- Attachments can be created, retrieved, updated, and deleted.
- Attachments are owned by exactly one task.
- Task-owned attachments are returned in a stable order.
- Parent/child task relationships remain reciprocal.
- Parent/child cycles are rejected.
- Blocking relationships are validated.
- Manual progress works.
- Computed progress works.
- Date-only deadlines and datetime deadlines are both supported.
- All datetime fields are stored and returned as UTC ISO 8601 strings.
- The backend does not perform local timezone conversion for display.
- All persisted entities receive short global Base62 IDs.
- Data persists across service restarts.
- OpenAPI describes every public API route.
- Tests pass.
